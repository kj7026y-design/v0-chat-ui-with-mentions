# StoryChat 프로젝트 개발 규칙

## 기본 원칙

* TypeScript와 함수형 React 컴포넌트를 사용한다.
* 기존 코드 구조, 네이밍, 스타일을 우선한다.
* 요청과 관련 없는 코드나 파일은 수정하지 않는다.
* 기존 동작을 유지하면서 필요한 범위만 수정한다.
* 동일한 기능의 기존 컴포넌트, 함수, 훅, 저장소가 있으면 재사용한다.
* 새로운 라이브러리는 기존 의존성으로 해결할 수 없을 때만 추가한다.
* 실행 가능한 완성 코드로 작성하고 임시 코드나 TODO를 남기지 않는다.

## 프로젝트 구조

현재 루트 구조를 유지한다.

```text
app/          페이지, 레이아웃, API Route
components/   공용 및 기능별 React 컴포넌트
components/ui 공용 UI 컴포넌트
hooks/        공용 React 훅
lib/          상태, 저장소, 채팅 엔진, 유틸리티
lib/rp/       AI 롤플레잉 파이프라인
lib/server/   서버 전용 코드
public/       정적 파일
```

* 별도 요청 없이 `src` 구조로 변경하지 않는다.
* 페이지는 `app/<route>/page.tsx`에 작성한다.
* API는 `app/api/<route>/route.ts`에 작성한다.
* 기능 컴포넌트는 `components/chat`, `components/create`, `components/timeline` 등 기존 기능 폴더에 배치한다.
* 서버 전용 코드는 `lib/server`에 배치하고 클라이언트에서 import하지 않는다.
* AI 프롬프트와 모델 처리 코드는 `lib/rp` 구조를 유지한다.

## 네이밍과 TypeScript

* 파일과 폴더 이름은 소문자와 하이픈을 사용한다.
* 컴포넌트는 PascalCase, 함수와 변수는 camelCase를 사용한다.
* 불리언 변수는 `is`, `has`, `can`, `should`로 시작한다.
* 컴포넌트와 일반 함수는 named export를 우선한다.
* Next.js 페이지와 레이아웃은 default export를 사용한다.
* 객체 구조는 `interface`, 유니온과 리터럴 타입은 `type`을 사용한다.
* `enum` 대신 `as const` 객체를 사용한다.
* `any`, `as any`, 불필요한 타입 단언을 사용하지 않는다.
* 함수 인수가 많으면 객체 매개변수를 사용한다.

## React 규칙

* 클래스 컴포넌트를 사용하지 않는다.
* 이벤트, 브라우저 API, 로컬 상태, Zustand가 필요한 경우에만 `"use client"`를 사용한다.
* props나 기존 상태에서 계산 가능한 값을 별도 state로 저장하지 않는다.
* 값 계산을 위해 `useEffect`를 사용하지 않는다.
* Effect의 이벤트, 타이머, 요청에는 cleanup을 작성한다.
* 중첩 삼항 연산자와 깊은 조건문을 피하고 조기 반환을 사용한다.
* 배열 인덱스를 React key로 사용하지 않는다.
* 성능 근거 없이 `useMemo`와 `useCallback`을 남용하지 않는다.

## 상태와 저장소

* 컴포넌트 내부 UI 상태는 `useState`를 사용한다.
* 여러 화면에서 공유하는 상태는 기존 Zustand 스토어를 사용한다.
* 새로운 전역 상태 라이브러리를 추가하지 않는다.
* 브라우저 저장은 기존 `*-storage.ts` 모듈 패턴을 사용한다.
* 컴포넌트에서 동일한 `localStorage` 키를 직접 중복 관리하지 않는다.
* 서버 렌더링 중 `window`와 `localStorage`에 접근하지 않는다.
* 저장 데이터 복원 시 hydration 오류를 방지한다.

## UI와 스타일

* 기존 `components/ui`의 공용 컴포넌트를 우선 재사용한다.
* 동일한 버튼, 모달, 입력창, 드로어를 중복 생성하지 않는다.
* 아이콘은 기존 Lucide React 사용 방식을 따른다.
* 알림은 기존 모달 또는 Sonner toast 패턴을 따른다.
* 스타일은 Tailwind CSS를 우선 사용한다.
* 조건부 클래스는 기존 `cn` 유틸리티를 사용한다.
* 전역 스타일과 디자인 토큰은 `app/globals.css`를 기준으로 한다.
* 기존 화면의 색상과 디자인을 요청 없이 전체 변경하지 않는다.
* 수정하는 파일의 기존 따옴표와 세미콜론 스타일을 유지한다.
* 포맷 변경만으로 파일 전체 diff를 만들지 않는다.

## 모바일 채팅 UI

* 모바일 화면을 우선하고 데스크톱으로 확장한다.
* 모바일 높이는 `100dvh` 사용을 우선 검토한다.
* 헤더, 상태 영역, 메시지 목록, 입력창, 컨텍스트 박스, 하단 내비게이션의 높이와 겹침을 함께 확인한다.
* 모바일 키보드가 입력창과 메시지를 가리지 않도록 한다.
* 불필요한 중첩 스크롤을 만들지 않는다.
* 고정 요소는 safe area와 z-index를 고려한다.
* 채팅 자동 스크롤이 이전 메시지를 읽는 사용자를 방해하지 않도록 한다.

## 입력과 API

* 사용자 입력은 클라이언트와 서버에서 모두 검증한다.
* API 요청 body는 검증 전까지 신뢰하지 않는다.
* API 검증은 기존 Zod 패턴을 우선 사용한다.
* 한글 IME 조합 중 Enter가 메시지를 전송하지 않도록 처리한다.
* 제출 중 중복 요청과 중복 메시지 생성을 막는다.
* API 오류는 적절한 HTTP 상태와 사용자 친화적인 메시지로 반환한다.
* API 키와 서버 환경 변수는 클라이언트에 노출하지 않는다.
* 개인정보, 인증 헤더, 전체 프롬프트를 로그에 출력하지 않는다.

## AI 채팅 엔진

다음 처리 단계를 UI 컴포넌트와 분리한다.

```text
입력 정규화
→ 배경과 컨텍스트 구성
→ 모델 설정 적용
→ 프롬프트 생성
→ AI 제공자 호출
→ 스트림 처리
→ 응답 검증
→ fallback 처리
→ 메시지 저장
→ UI 반영
```

* UI 컴포넌트에 시스템 프롬프트나 모델별 분기를 직접 작성하지 않는다.
* 입력 처리는 normalizer, 프롬프트는 prompt, 모델 차이는 providers 또는 model-profiles에서 관리한다.
* 반복 검사와 응답 검증은 validation에서 처리한다.
* 하나의 규칙을 여러 프롬프트 파일에 중복 작성하지 않는다.
* 부분 응답과 최종 응답을 구분한다.
* 요청 취소와 실제 오류를 구분한다.
* 스트리밍 중 메시지와 크레딧이 중복 저장 또는 차감되지 않도록 한다.

## 접근성과 품질

* 동작 요소는 `<button>`, 페이지 이동은 Next.js `Link`를 사용한다.
* 클릭 가능한 `<div>`를 만들지 않는다.
* 아이콘 버튼에는 접근 가능한 이름을 제공한다.
* 모달과 드로어는 키보드, ESC, 포커스 이동을 지원한다.
* 로딩, 오류, 빈 상태를 처리한다.
* 사용하지 않는 import, 디버깅 로그, 주석 처리된 코드를 남기지 않는다.
* 수정 후 타입 오류, import 오류, 모바일 레이아웃, 다크 모드, 기존 기능을 확인한다.
* 가능한 경우 lint와 build를 실행하고 실패한 검증은 명확히 알린다.

## 금지 사항

* 요청 없는 프로젝트 구조 변경
* 요청 없는 대규모 리팩터링
* 기존 UI 컴포넌트 전체 재생성
* 새로운 상태 관리 방식의 임의 도입
* UI 내부에 AI 프롬프트와 서버 로직 작성
* 클라이언트에서 비밀 키 사용
* `any`로 타입 오류 숨기기
* 관련 없는 파일의 포맷 변경
* 모델 오류를 정상 채팅 메시지로 저장
* 스트림 완료 전 메시지 확정 또는 크레딧 중복 차감
